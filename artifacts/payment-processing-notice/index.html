Pago en procesamiento como estado seguro del checkout.

La mejora no es solo cambiar el notice. Es definir qué debe ver y qué no debe poder hacer el user mientras el pago se autoriza, sin crear riesgo de loading infinito si la order falla.

Decisión propuesta

Cuando el user toca “Finalizar compra”, el checkout entra en un estado global de procesamiento.

El estado debe bloquear edición y doble submit, mostrar un mensaje claro de progreso y salir siempre por uno de tres caminos: éxito con redirect, error recuperable en checkout o failsafe por tiempo máximo.

Hoy vs dirección

Hoy: aviso local

El mensaje aparece debajo de “Forma de pagamento”. Puede parecer solo informativo, mientras el CTA y otros elementos siguen compitiendo por atención.

Ainda estamos processando sua compra. Falta pouco! Não feche nem atualize a página.
Dirección: estado del checkout

El checkout queda visualmente en espera. El user entiende que la compra está avanzando y que no debe interactuar hasta recibir resultado.

Procesando pago Espera unos segundos

Failsafe obligatorio

El loading nunca puede quedar bloqueando el checkout.

Si el user no fue redirigido y volvió al checkout porque ocurrió un error al finalizar la order, el componente de loading tiene que desaparecer. En ningún caso el nuevo diseño puede dejar al user bloqueado en una página con loading infinito.

  • Al recibir error de order/payment, remover el estado global de processing y mostrar el error recuperable.
  • Si no hay resultado después de un tiempo máximo, salir del loading y permitir recuperación. Sugerencia inicial para revisar con Engineering: 10 segundos.
  • El estado de processing no debe persistir por cache, reload, back navigation o re-render si el checkout ya volvió con error.

Reglas mínimas de comportamiento

Al iniciar procesamiento

Deshabilitar edición de dirección, envío, pago, cupón y CTA. Mostrar un mensaje global, no solo un banner dentro del componente de pago.

Mientras procesa

Mantener el contexto visible, pero atenuado. Evitar cualquier affordance que sugiera que el user puede finalizar de nuevo.

Si falla o demora

Salir del loading, devolver control al user y mostrar el próximo paso: reintentar, elegir otro método o corregir la información necesaria.

IdleUser puede revisar y editar checkout.
SubmitCTA inicia creación/autorización de order.
ProcessingCheckout bloqueado temporalmente.
ResultRedirect de éxito o error recuperable sin loading.

Criterio de aceptación

La solución solo es válida si mejora claridad sin empeorar recuperación.

El diseño puede evolucionar, pero el contrato mínimo debe quedar fijo: processing es un estado temporal, visible y cancelado automáticamente ante error o timeout. Si el user está de vuelta en checkout, el checkout tiene que estar usable.
